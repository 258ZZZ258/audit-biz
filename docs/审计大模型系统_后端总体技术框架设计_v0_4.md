# 审计大模型系统：后端总体技术框架设计方案（V0.4 草案）

> 适用项目：东方证券审计大模型系统一期
> 覆盖范围：后端总体工程架构 —— 服务拓扑与职责边界（audit-biz / audit-ai）、数据库与中间件选型、网关方案（模型网关 / API 网关）、鉴权与权限体系、服务间通信契约、异步编排、外部数据源接入、可观测与部署。**架构经甲方确认：与前端页面交互的部分一律 Java（audit-biz）；智能体与文档解析保留 Python（audit-ai）**。
> 与既有文档关系：本文档是《制度查询与制度比对智能体_RAG技术框架设计》V1.5（下称"RAG 设计"）与《文档处理与语料库构建_技术框架设计》V1.6（下称"管线设计"）的**工程承载层总览**；前两者定义"AI/RAG 内部做什么"，本文档定义"整套后端如何搭、谁调谁、数据落在哪、权限怎么过"。**不重复**两者已锁定的检索/比对/摄取内部细节。
> 文档性质声明：【标准】= 行业成熟实践；⚠ = 合成启发式/工程判断，须实测标定；**【待甲方确认】= 字段口径/协议/配额需与甲方书面确认后冻结**。本文档须经张翼飞复核后方可作为对甲方汇报底稿。
> V0.2 增量（CP-009，源自第二次功能点会议）：① 文件版本语义拆分（`file_versions` ≠ 制度 `doc_versions`，§4.7）；② 通用文件/资料管理子系统（功能 10 基础支撑落地：上传/查询/下载/删除/版本/标签库/删除守卫，§4.7）；③ 生成物生命周期与状态机（草稿待确认/已确认/需重新生成，跨监管报告/审计规范/费用报告/审计报告，§4.8）；④ 所有权（owner）行级 ABAC（"本人导入""生成者权限最大"，§7.x）。功能专项细节（审计规范模板映射 / 费用数据来源监控 / 报告格式检查引擎）作为输入移交 CP-008 §13.6 规划的各功能独立文档，本文档仅保留数据模型挂钩与边界声明。
> **V0.3 已作废**：纯 Python 单后端方案因甲方硬约束"前端交互层必须 Java"而不成立，相关内容不再适用。
> **V0.4 增量（CP-011，回到双语言基座 + 三刀简化）：在 v0.2（Java 前端交互层 + Python 智能体/解析）基础上做三处降复杂度简化——① Java 栈定为最小常规栈（Spring Boot MVC + MyBatis-Plus + jCasbin + 最小化 Spring Security resource-server + Apache POI），授权归 Casbin、不用 Spring Security 授权层；② 把 audit-ai 从"对等服务"收缩为"无身份、无前端契约的无状态推理/解析端点"，删除内部带身份 JWT（边界二），Java 把预计算的权限过滤值当普通字段下传，内部边界压成"单向、无身份、共享密钥 + 网络隔离"；③ Phase 1 用 PG 状态机替代 Temporal（信创内网可部署性未验证的 P0 风险，v1.6 已留兜底），少维护一个高风险中间件。SSO 已确认走标准协议，认证方案随之收口（§7）。**

## 0. 设计范围与定位

本文档覆盖一期八项核心功能的**共用后端骨架**与各功能的后端落位，不含前端（Vue + genesis-ui）实现细节，不含 RAG/比对/摄取的算法内部（见 RAG 设计、管线设计）。

**与既有文档的边界划分：**

| 关注点 | 归属文档 | 本文档是否展开 |
|---|---|---|
| 混合检索 / 重排 / 路由 / 覆盖判定算法 | RAG 设计 §5–§8 | 否，仅引用接口契约 |
| 摄取管线 S0–S5 / 解析 / 切块 / 质检 | 管线设计 §3–§9 | 否，仅引用数据模型 |

| **数据库与中间件选型 / 数据模型全景** | 本文档 §3–§5 | 是 |
| **网关 / 鉴权 / 服务间契约 / 错误码** | 本文档 §6–§8 | 是 |

## 1. 总体架构与服务拓扑

延续"1 平台 + 多形态应用"原则。后端按甲方约束分两层：**与前端交互的一切走 audit-biz（Java）；智能体与文档解析保留 audit-ai（Python）**。两套语言是甲方强制、省不掉的；本版的简化集中在**让两者之间的边界尽量薄**：

```
┌────────────────────────────────────────────────────────────────────────┐
│ 用户/前端  audit-web (Vue3 + genesis-ui)                                 │
│   制度查询(SSE流式) │ 比对/报告/费用(任务提交+状态轮询) │ 报表/导出页面     │
└───────────────────────────────┬────────────────────────────────────────┘
                                 │ HTTPS (甲方SSO令牌 + 我方API)
┌───────────────────────────────▼────────────────────────────────────────┐
│ audit-biz (Java / Spring Boot MVC)  —— 唯一对外入口 + 全部有状态的东西     │
│  最小常规栈: Spring Boot MVC + MyBatis-Plus + jCasbin                     │
│           + 最小化 Spring Security(仅 resource-server 验令牌) + POI       │
│  · 鉴权(SSO标准协议验令牌) + 授权(jCasbin, 六类权限点) + 操作/审计日志      │
│  · 业务数据CRUD、费用规则引擎(SQL)、报告/Excel/Word渲染、文件/资料管理      │
│  · 任务提交与状态(PG状态机, Phase1)、引用四级回查装配、外部数据源接入       │
└──────────┬─────────────────────────────────────────────────────────────┘
           │ 单向 REST/JSON + 共享密钥 + 网络隔离 (无身份, Java 传预计算过滤值)
           │ Java 调一下就完；SSE 由 Java 透传 Python 流
┌──────────▼─────────────────────────────────────────────────────────────┐
│ audit-ai (Python / FastAPI)  —— 无状态推理/解析端点(内网仅对 audit-biz 可达)│
│  · 在线: /retrieve /generate /compare  (无身份, 按传入过滤值检索)          │
│  · 离线: /ingest-batch /ocr  文档解析管线 S0–S5(写 PG 语料表 + Milvus)     │
│  · 经甲方模型网关调 LLM/Embedding/Rerank；不持有对前端的契约               │
└──────────┬──────────────────┬──────────────────────────────────────────┘
           ▼                  ▼
┌────────────────────────────────────────────────────────────────────────┐
│ 基础设施(我方自建)  PostgreSQL(权威, 兼任务状态机) │ Milvus2.4 │ MinIO     │
│                    (Temporal 推迟到规模需要时再上) │ Langfuse │ Redis(可选⚠)│
├────────────────────────────────────────────────────────────────────────┤
│ 甲方提供基础设施   模型网关(OpenAI兼容,强制) │ 接口网关/SSO(标准协议)/组织权限 │
├────────────────────────────────────────────────────────────────────────┤
│ 外部数据源(单向只读) 智能合规 │ 全面预算 │ 费控 │ 监督信息共享平台 │ OA(待定) │
└────────────────────────────────────────────────────────────────────────┘
```

**核心原则**：① audit-biz 吃掉所有有状态的东西（数据、权限、任务、PG、对前端的一切契约），是唯一鉴权与对外边界；② **audit-ai 收缩为无状态端点**——不认身份、不做权限判断、不持有前端契约，在线只跟 Milvus 和模型网关说话（离线解析管线仍写 PG 语料表与 Milvus，这是它作为语料生产者的固有职责）；③ 两者之间是**单向、无身份**调用：Java 把预计算好的权限过滤值（perm_tags/分区/project_id）当普通字段传下去，用共享密钥 + 网络隔离即可，**不再签发带身份的内部 JWT**。这样 v0.2 里最重的跨服务身份传递、双策略库同步、引用回查归属扯皮全部消失，而不触碰任何红线。

## 2. 服务划分与职责边界

边界划分原则：**与前端交互的有状态能力全部在 Java（audit-biz）；智能体推理与文档解析在 Python（audit-ai），且 audit-ai 收缩为无状态端点**。

| 能力 | audit-biz (Java) | audit-ai (Python) | 说明 |
|---|---|---|---|
| 对外 API 入口 / 鉴权 / 授权 | ✅ | — | 唯一入口；SSO 验令牌（Spring Security resource-server）+ jCasbin 授权 |
| RAG 检索 / 重排 / 路由 / 生成 | — | ✅ | RAG 设计 §5–§6；无身份，按传入过滤值检索 |
| 文档比对（义务抽取/覆盖判定） | — | ✅ | RAG 设计 §7 |
| 文档解析摄取管线 S0–S5 | — | ✅ | 管线设计；离线写 PG 语料表 + Milvus（语料生产者固有职责） |
| 费用规则引擎（5 维 SQL 判定） | ✅ | — | 跑在业务表上；LLM 仅辅助 |
| 费用 OCR + 违规原因说明 | 调用 | ✅ 计算 | Java 发起，Python /ocr 返回文本 |
| 报告/Excel/Word 模板渲染 | ✅ | — | Apache POI/docx4j（含报告格式检查，Java 强项） |
| 业务数据 CRUD（费用/监督/项目/模板/机构人员/文件资料） | ✅ | — | 业务表权威在 Java 写 |
| 操作日志 / 审计日志 | ✅ | — | 全量写 PG（启动会权限验收） |
| **引用四级定位回查装配** | ✅ 回查装配 | 只返回 chunk_id/clause_id | **v0.4 翻转**：Java 独占 PG 与前端契约，由它回查填充（§8.2） |
| 任务提交与状态（前端轮询） | ✅ | 上报状态 | PG 状态机（§5），Phase 1 不用 Temporal |
| 外部数据源接入/同步 | ✅ | — | 业务联系单接口；只读落业务表 |

**Java 栈（最小常规栈，交接友好）**：Spring Boot **MVC**（不用 WebFlux，避免响应式给小团队加认知负担）+ **MyBatis-Plus**（费用规则是 SQL 重活，比 JPA 透明，信创 Java 圈事实标准）+ jCasbin + 最小化 Spring Security（仅 resource-server 验令牌，§7）+ Apache POI/docx4j。砍掉 Spring Cloud 微服务全家桶、WebFlux、独立 API 网关产品（BFF 就是这个 Spring Boot 应用本身）。

**audit-ai 无状态说明**：在线端点（/retrieve /generate /compare /ocr）不认身份、不持前端契约、不做权限判断——Java 把算好的过滤值（perm_tags/分区/project_id）当普通字段传入，Python 据此构造 Milvus filter（"检索前置过滤"红线照样满足）。离线端点（/ingest-batch）是文档解析管线，写 PG 语料表 + Milvus，这是它作为语料生产者不可避免的 PG 写入，与"在线无状态"不矛盾。

**规则引擎归属说明**：费用 5 维规则是对 expense_records/budget_lines/approval_records 等业务表的 SQL 判定（CP-008 §13.3），落在 Java；audit-ai 仅在"发票 OCR + 违规原因解释"被 Java 调用。

## 3. 数据库选型

延续管线设计"PG 为权威事实库、Milvus 为可重建检索投影"的一句话架构，不改变。

| 存储 | 选型 | 角色 | 信创/工程理由 |
|---|---|---|---|
| 关系库 | **PostgreSQL** | 权威事实库（制度/版本/条款/比对/案例/业务表族/权限/日志），兼 demo 状态机、图谱初期递归 CTE | 【标准】事务一致、JSONB 灵活、递归 CTE 免单独图库；国产化有人大金仓/openGauss 兼容路径【待甲方确认】具体发行版 |
| 向量库 | **Milvus 2.4 单机** | 检索投影（dense+sparse 混合 + RRF），可随时从 PG 全量重建 | RAG 设计 §4.1 已锁；upsert 幂等由确定性 chunk_id 保证 |
| 对象存储 | **MinIO** | 原件 / IR / 导出文件 / 发票附件 / 模板原件 | 管线设计 §12.4 已用；S3 兼容、可离线部署 |
| 缓存/会话 | **Redis（可选）⚠** | 会话态、限流计数、分布式锁、热点字典缓存 | **非既有文档项**；一期 ~100 并发量级下可暂不引入，demo 阶段用 PG 替代，遵循"最小脊柱、第二消费者触发抽取"纪律。是否启用见 §13【待甲方确认】部署环境是否提供 Redis |

**一致性模型（复用管线设计 §12.1，不改）**：写顺序 PG → Milvus upsert → flush → PG 置 INDEXED；最坏态为"PG 有、Milvus 无"的检索不可见半成品，重试收敛，不存在脏向量。业务表族（费用/监督/项目）为纯 PG 事务，不投影 Milvus（发票图像 OCR 文本除外，入 `corpus_type=expense_doc` 分区）。

## 4. 数据模型全景（PG，add-only）

复用管线设计 §10 与 RAG 设计 §4.2 已定义的全部表（import_batches / documents / doc_versions / clauses / chunks / regulation_mapping / obligation_keywords / comparison_tasks / comparison_findings / cases / clause_references / 各 dict_* 字典表）。本文档**新增**后端共用与各场景业务表族，全部 add-only、带 `created_at/by, updated_at/by`、敏感操作额外写审计日志表。

### 4.1 权限与组织（从甲方 SSO/组织系统同步的只读镜像）

| 表 | 用途 | 待确认项 |
|---|---|---|
| `users` | 用户镜像（user_id、姓名、所属机构、职级） | 【待甲方确认】SSO 用户字段 schema、同步方式（推/拉/增量） |
| `org_units` / `user_org` | 组织机构树 + 用户归属 | 【待甲方确认】组织层级口径、与"机构人员"功能 10 的关系 |
| `roles` / `casbin_rule` | 角色定义 + Casbin 策略存储（jCasbin 适配器表） | 【待甲方确认】角色清单与权限矩阵（启动会交付物，M4 冻结） |

> 权限分级最小三级：普通员工 / 主管 / 总经理（RAG 设计制度查询 I/O 要求）。组织/用户为**只读镜像**，不在本系统维护账号。

| `template_fields` | 字段映射 schema（占位符 → 业务字段） | 【待甲方确认】各场景字段映射（开发前冻结，启动会"模板风险"） |
| `template_usage_logs` | 使用留痕 | — |

### 4.7 文件/资料管理子系统（功能 10 基础支撑落地，CP-009 新增）

面向**通用资料**（"上传资料 / 知识资料 / 被审计单位资料"等，非制度语料）的上传/查询/下载/详情/版本/删除/标签管理。与管线的**语料摄取**（制度/外规/案例走 P-INT/P-EXT/P-CASE）是两套入口：语料摄取产出 chunks 供检索，本子系统只做文件级管理，落在 audit-biz（Java）。

| 表 | 用途 | 待确认项 |
|---|---|---|
| `resource_files` | 通用资料登记（file_id、使用单位、资料标题、资料所属时间、资料类型、标签数组、密级、owner=created_by、当前版本引用、引用计数） | 【待甲方确认】通用上传字段是否够、与制度类 manifest（发文字号体系）如何并存 |
| `file_versions` | **文件级版本**（同名上传→新版本），与 `doc_versions`（制度修订版本）严格区分 | 见下方版本语义边界 |
| `tags` | 标签库（tag_id、名称、分类、dict_version），支持增删改查 | 【待甲方确认】受控字典 vs 自由打标 |
| `file_references` | 文件被引用登记（file_id ↔ 引用方类型/ID：生成报告/比对任务/导出/规范），删除守卫依据 | 【待甲方确认】"被引用"判定范围 |

**版本语义边界（CP-009 关键澄清，须 CP 回灌 V1.6）**：
- `doc_versions`（管线 §7.2）= **制度修订版本链**：须 manifest 声明替代文号、录修订说明、做新旧版 diff，并触发比对 Pass 1 与审计规范变更打标队列。**仅制度/外规走此**。
- `file_versions`（本子系统）= **通用资料文件级版本**：同名上传即新版本，**无修订语义、不触发任何制度流转、不进比对队列**。
- 两者**不可混用**。通用资料若后续被认定为制度类，须经显式转换流程，不得自动并入 `doc_versions`。混用会把普通资料误挂进制度修订流转、污染比对 Pass 1。

**删除守卫（"被引用的文件不能删除"）**：删除前查 `file_references` 引用计数，count>0 → 拒绝并返回引用方清单；仅 owner 可删除**未引用**误传文件；删除为软删 + MinIO 原件保留（承袭管线 §12.3 留证不删）。引用计数在生成报告/比对/导出/规范引用文件时 +1、引用方删除时 -1（事务内维护）。

**权限**：默认按"职级可见范围 ∪ 本人导入（owner）"双维度（§7.x 所有权 ABAC），可细化。

### 4.8 生成物生命周期与状态机（CP-009 新增）

统一登记跨场景生成物（监管报告/审计规范/费用报告/审计报告），避免各功能散表、状态语义不一。

| 表 | 用途 | 待确认项 |
|---|---|---|
| `generated_artifacts` | 生成物登记（artifact_id、类型、状态、当前版本、owner=生成者、关联场景参数：时间段/业务范围/机构/**监管部门**） | — |
| `artifact_versions` | 生成物版本（记录修改内容 + 时间；新版本 vs 覆盖当前版本规则） | 【待甲方确认】重生成默认新版本还是覆盖当前 |

**状态机**：`draft_pending`（草稿待确认）→ 人工确认 → `confirmed`（已确认）；关联条件变化（如所依据制度变更）→ 置 `needs_regen`（是否需重新生成）标记。状态机不回写任何源系统（守红线）。

- **时间维度统一用"时间段"**（start_date/end_date），不存具体时间点（会议明确"时间周期改为时间段，避免具体时间"）；监管报告新增 `监管部门` 字段。

- 审计规范的"优先基于 base 版本差异比对、0-1 生成先手动选法规+数据再人工干预"——**算法细节属功能专项**，移交《审计规范智能化》文档；本表仅提供版本与状态承载。

> **数据字段字典种子**：以上全部表的字段名/类型/枚举/必填/来源系统，直接导出为启动会交付物《数据字段字典》，是 M4 接口模板冻结的输入。

## 5. 中间件选型

### 5.1 异步编排：Phase 1 用 PG 状态机（Temporal 推迟）

**简化决策（CP-011）**：Phase 1 **不引入 Temporal**，改用 **PG 状态机 + 轮询 worker**（管线设计 §11 早已留作兜底）。理由：Temporal 在信创内网的可部署性是未验证的 P0 风险；Phase 1 只有"文档解析摄取"与"制度比对"两类异步，PG 状态机 + 单 worker 轮询足够；少维护一个高风险中间件，同时 Java 侧也不必引入 Temporal Java SDK。Activity 仍要求幂等、瞬时错误退避重试、大批次分片续跑。**离线任务优先级低于在线问答**（经模型网关统一限流，CP-005）。Temporal 留到规模或编排复杂度真正需要时再上（届时起 CP）。

### 5.2 任务状态聚合

Java（audit-biz）作为唯一任务提交入口与状态面：任务记录写 `async_tasks`（含身份+数据范围快照），Python worker 执行并回报进度，前端只轮询 Java 一个面。

| 工作流 | 执行方 | 主导工作 |
|---|---|---|
| 文档解析摄取（IngestBatch） | audit-ai (Python worker) | 解析/向量化，写 PG 语料表 + Milvus |
| 制度比对（Comparison） | audit-ai (Python worker) | LLM 覆盖判定，结果回写 comparison_findings（经 Java 或直写，见下） |
| 审计报告 / 费用批量 / 监督月度（Phase 2–3） | audit-biz 主导 + 调 audit-ai | 业务/规则为主 + 段落/OCR 辅助 |

> 比对/解析结果落库：为守"audit-ai 在线无状态"，**结果回写由 Java 收口或经专用离线写入通道**（离线管线本就写 PG 语料表，属固有职责，不算在线状态）。`async_tasks` 单写入方（Java），状态聚合无跨语言一致性问题。

## 6. 网关方案

对外只有**两类真正的网关**（原"内部服务调用"在 v0.4 压成单向无身份调用，不再是网关边界）：

### 6.1 甲方模型网关（LLM 算力侧，强制，CP-005）

- 【待甲方确认】embedding endpoint 是否暴露 dense+sparse 双输出、版本钉死与变更通知（CP-005-②）
- 【待甲方确认】MCP/SKILL 工具（检索/比对/cases SQL）注册协议与鉴权方式（P0 #5，杨老师）

### 6.2 我方 API 网关 / BFF（audit-biz 承担）

一期单业务服务，**不另起独立网关进程**，由 audit-biz（Spring Boot **MVC**）充当唯一入口与 BFF。职责：
- **鉴权**：对接甲方统一认证（SSO 已确认走标准协议）。用**最小化 Spring Security（仅 resource-server 子集）**验令牌、解析当前用户；表单登录/session/方法级安全全关，过滤链 permitAll（§7）
- **授权**：jCasbin（菜单/功能/导出/模板/规则），**不用 Spring Security 授权层**
- **限流**：Resilience4j / bucket4j 本地限流 + 模型网关配额二级保护
- **SSE 透传**：制度查询为流式，audit-biz 用 `SseEmitter`（Spring MVC）透传 audit-ai 的流式输出到前端
- **错误码统一**：对外错误码规范由 audit-biz 收口（§8.3）
- 【待甲方确认】SSO 最终是哪个协议（OAuth2/OIDC/CAS/SAML/JWT）——决定用哪个 Spring Security 子模块（§7、§12）

### 6.3 内部调用（audit-biz → audit-ai，单向无身份）

内网 REST/JSON，**audit-ai 仅对 audit-biz 网络可达**。**不再有内部带身份 JWT**：Java 把预计算好的权限过滤值（perm_tags/分区/project_id）当**普通请求字段**传入，用**共享密钥 + 网络隔离**做服务间认证即可（§7）。这是 v0.4 相对 v0.2 删掉的最大一块复杂度。

5 个外部数据源经甲方接口网关 / 业务联系单接口接入（信息技术部门 owns 接口网关），单向只读（§10）。

## 7. 鉴权与权限体系

两条边界（v0.2 的"内部带身份 JWT"边界在 v0.4 被简化掉）：

**边界一 用户↔甲方SSO（认证）**：SSO 已确认支持标准协议（OAuth2/OIDC/CAS/SAML/JWT），认证方案随之收口为**最小化 Spring Security（仅 resource-server 子集）**：
- OAuth2 / OIDC / JWT → `spring-boot-starter-oauth2-resource-server`，配 issuer/JWKS 自动验签
- SAML → `spring-security-saml2-service-provider`
- CAS → Spring Security CAS 客户端
- 过滤链关掉表单登录/session/方法级安全，一律 permitAll；Spring Security **只留"验令牌 + 解析当前用户"这一件事**

> 为何不手写认证过滤器：既然是标准协议，resource-server 自动验签比自撸 JWKS 轮转/签名校验更省更稳，且券商安全审查更认 Spring Security 标准件 ⚠。具体用哪个子模块取决于甲方最终给的协议（§12 待确认）。

**授权（关键简化）**：六类权限点（菜单/数据/文件/导出/模板/规则）**全部归 jCasbin**，**不用 Spring Security 授权层**（`@PreAuthorize`/方法级安全对本项目是死重，授权权威是 Casbin）。jCasbin 是唯一 enforcer，策略存 PG `casbin_rule`。

**边界二 audit-biz→audit-ai（单向无身份，已大幅简化）**：audit-ai **不认身份、不做权限判断**。Java 在 jCasbin 算出用户的数据范围后，把**预计算的过滤值**（perm_tags 清单 / 分区 / project_id）当**普通请求字段**传给 Python；Python 直接据此构造 Milvus filter。
- **检索前置过滤红线照样满足**（RAG 设计 §5.5）：过滤位仍在检索阶段生效，只是"算"在 Java、"用"在 Python
- **删除项**：内部 JWT 签发/校验、HS256/RS256 取舍、TTL、claim 设计、身份属性下沉——全部不需要
- 服务间认证：共享密钥（配置注入）+ 网络隔离（audit-ai 仅对 audit-biz 可达）

**边界三 MCP/SKILL 工具调用**：网关回调工具的鉴权独立，协议待杨老师确认（P0 #5）。

**异步任务身份快照**：长任务（比对 30–120min）由 Java 在提交时把身份+数据范围快照固化为任务入参（写 `async_tasks`），worker 用服务身份 + 快照过滤值跑——避免长任务期间会话过期。

### 7.x 所有权（owner）行级 ABAC（CP-009 新增）

除密级/职级 `perm_tag` 外，**通用资料与生成物**引入 `created_by`（owner）行级维度，落实会议两条权限语义：
- **"按职级 + 本人导入情况"**：`resource_files` 查询默认可见范围 = 职级可见 ∪ 本人导入（owner=当前用户）；可由权限矩阵细化。
- **"生成文件的人权限最大"**：`generated_artifacts` 的 owner 可读写/删/重生成，他人按职级降级为只读或不可见。

落点：通用资料/生成物的管理页查询为 **audit-biz 侧 SQL 行级过滤**（owner 字段 + 职级）；仅当资料进入 audit-ai 检索路径时（如 `corpus_type=audit_project` 项目资料联合检索），owner/project_id 作为 Milvus 标量过滤位与 perm_tag 并列前置过滤。**owner 维度不作用于制度语料检索**（制度无个人归属语义），避免误伤共享知识库。

> 权限点口径：RAG 设计 §8.3 写"六类"但仅列五项，启动会权限验收明列六项含"规则"。以启动会为准，建议 CP 回灌 V1.5（§15）。

## 8. 服务间通信契约

### 8.1 同步（制度查询 Chatbox）

前端 → audit-biz（`SseEmitter` 透传）→ audit-ai（流式）。audit-ai 产出**答案块 + 结构化引用标识**（chunk_id/clause_id + confidence + ai_label + route_type），**audit-biz 回查 PG 装配最终对前端契约**（RAG 设计 §5.9 字段：`answer_blocks[] + citations[](clause_id→doc→page→version→status) + ...`）。判定型路由（route_type=judgmental）前端渲染"AI 辅助判断，人工复核"框。

### 8.2 引用四级定位回查归属（v0.4 定稿：Java 收口）

**由 audit-biz 回查 PG 填充完整 citation 对象**——Java 独占 PG 与对前端的契约，audit-ai 只返回轻量的 chunk_id/clause_id。这样 audit-ai 在线热路径**不依赖 PG 回查**（无状态），v0.2"谁回查"的跨服务协调彻底消解，也不再需要回灌 V1.5 协调。

### 8.3 错误码体系

复用管线设计 §11.2 摄取侧 E1xx–E8xx；新增**业务/API 段**：
- `B1xx` 鉴权/权限（B101 未认证、B102 越权、B103 token 失效、**B104 内部令牌无效**——服务向:audit-biz↔audit-ai 共享密钥校验失败,由边界契约 `SPEC-BOUNDARY` 引入回灌）
- `B2xx` 业务校验（费用/监督/项目/模板入参）
- `B3xx` 外部数据源接入（接口不可用、字段缺失、增量冲突）
- `B4xx` 任务编排（提交失败、状态非法迁移）

> 【待甲方确认】对外错误码规范须在技术联调会冻结（启动会技术联调会议程含"鉴权、错误码"）。

### 8.4 状态轮询

比对/报告/费用/导入等长任务，前端轮询 `GET /tasks/{task_id}` → audit-biz 读 `async_tasks` 返回（排队/处理中 x/y/完成/部分失败）。
